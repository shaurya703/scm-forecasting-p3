# SCM Forecasting Accuracy (P3)

End-to-end demo that quantifies the business impact of replacing biased
manual demand forecasts with an ML-based forecast. A synthetic but realistic
3-year dataset (30 SKUs × 4 Indian warehouses × 156 weeks) flows through a
SQLite warehouse, a walk-forward-backtested gradient-boosting model, and a
Streamlit executive dashboard.

> Research / teaching prototype. The dataset is synthetic — numbers are
> illustrative, not drawn from any real company.

---

| Metric | Manual forecast | ML forecast | Improvement |
|---|---:|---:|---:|
| MAPE (mean absolute % error) | **29.6 %** | **18.3 %** | −11.3 pp (−38.1 %) |
| Stockout events (3 yrs) | 1,162 | — | — |
| Lost revenue from stockouts | ₹13.59 Cr | — | — |
| Worst category (manual) | Sports & Outdoor (36.2 %) | | |
| Worst SKU (manual) | Ski Gloves @ Mumbai (59.4 %) | | |

These are the numbers that anchor the executive-summary tiles at the top of
the dashboard.

## 3. Quick start

```bash
# 1. Build the warehouse (creates forecasting.db with 3 yrs of weekly data)
python 02_generate_data.py

# 2. Sanity-check row counts, key constraints, and headline metrics
python 03_verify_data.py

# 3. Train the ML model (walk-forward backtest + 12-week future forecast)
python 04_train_ml_model.py

# 4. Launch the dashboard
streamlit run 05_dashboard.py
```

Steps 1–3 are idempotent — rerun any time to regenerate the DB from scratch.
The whole pipeline runs in well under a minute on a modern laptop.

## 5. Data model

The schema is organised in three layers — see `DATA_DICTIONARY.md` for the
full ERD and column-by-column reference.

**Reference / master data** — `categories`, `products` (30 SKUs),
`warehouses` (Bengaluru, Mumbai, Delhi, Kolkata).

**Forecasting facts**
- `sales_history` — weekly actuals (18,720 rows, grain = SKU × DC × week).
- `manual_forecasts` — planner baseline, carries a SKU-specific bias of
  −30 % to +45 % (the problem the project addresses).
- `ml_forecasts` — model output, schema supports multiple models via
  `model_name`.

**Downstream impact**
- `inventory_snapshots` — weekly stock position driven by forecast-based
  replenishment.
- `stockout_events` — every week on-hand stock couldn't cover demand,
  with `shortage_units` and `lost_revenue` in INR.

**`v_forecast_accuracy`** — the dashboard's main fact object. Pre-joins
actuals with both forecasts and derives `*_bias_units`, `*_abs_error`,
`*_mape` so tiles and charts become simple `GROUP BY`s.

## 6. ML pipeline (`04_train_ml_model.py`)

- **Model.** `sklearn.HistGradientBoostingRegressor`, one global model
  trained across all SKU × warehouse series (handles NaNs natively, trains
  in ~2 s per cycle).
- **Features.** Lagged demand (1 / 2 / 4 / 13 / 52 weeks), rolling means,
  calendar (week-of-year, month, quarter, year), static product / DC
  attributes, and the promotion flag.
- **Backtest.** Expanding-window walk-forward over the last 52 weeks,
  retraining every 13 weeks — realistic SCM planning cadence.
- **Prediction intervals.** Empirical 85 % bands from backtest residuals.
- **Outputs.** Backtest predictions written to `ml_forecasts`; a 12-week
  recursive future forecast written to `future_forecasts`; per-category
  accuracy report printed to stdout.

## 7. Dashboard (`05_dashboard.py`)

Four tabs, one file, no backend beyond SQLite:

1. **Executive Overview** — headline KPIs, manual-vs-ML lift, ₹ value at stake.
2. **Forecast Explorer** — per-SKU deep-dive: actuals vs both forecasts,
   with prediction bands on the ML line.
3. **Category & DC Analysis** — where the errors and losses concentrate.
4. **Future Forecasts** — next 12 weeks ML forecast with 85 % bands.

```bash
streamlit run 05_dashboard.py
# open http://localhost:8501
```

If port 8501 is taken: `streamlit run 05_dashboard.py --server.port 8600`.

## 8. Data generation assumptions

Documented in full in `DATA_DICTIONARY.md §4`. Summary:

- Time range 2023-01-02 → 2025-12-29 (156 Monday-anchored ISO weeks).
- Demand model: `actual = base × seasonality × trend × lognormal_noise × promo_mult`.
- Seasonality: annual sine wave, amplitude and peak week differ per SKU
  (winter jackets peak wk 2, swimwear wk 18, etc.).
- Trend: linear, −8 % to +20 % per year per SKU.
- Promotions: ~3 % of weeks, 2× – 3.5× demand uplift.
- Manual forecast bias: SKU-specific, −30 % to +45 %.
- Market multipliers: Mumbai 1.20×, Delhi 1.15×, Bengaluru 1.00×, Kolkata 0.70×.
- RNG seed: `np.random.seed(42)` — runs are fully reproducible.

## 9. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `no such table: sales_history` | step 1 hasn't run | `python 02_generate_data.py` |
| `no such table: future_forecasts` in dashboard | step 3 hasn't run | `python 04_train_ml_model.py` |
| `externally-managed-environment` from `pip` | installing into system Python | use a `venv`: `python -m venv .venv && source .venv/bin/activate` |
| Streamlit port 8501 taken | another app bound it | `streamlit run 05_dashboard.py --server.port 8600` |
| Numbers differ from README | different RNG seed / edited generator | restore `np.random.seed(42)` and regenerate |

## 10. Limitations & next steps

- **Synthetic data** — the "ML beats manual" lift is structural, not
  empirical. Plug real POS / shipment data into `sales_history` and
  rerun for meaningful numbers.
- **Single model** — only HistGradientBoosting. The `ml_forecasts` schema
  already supports multiple models via `model_name`; adding Prophet or
  LightGBM is a new training script, not a schema change.
- **Univariate drivers** — no price elasticity, cannibalisation, or
  cross-SKU effects. Promo is a flag, not a treatment.
- **No cost model** — `lost_revenue` is gross; true cost of stockouts
  should also include goodwill and substitution economics.
